Object-Oriented Programming organizes code around objects rather than a sequence of functions. A class is a blueprint, the way an architectural plan describes a house before it's built. An object is the actual thing built from that blueprint — the real house.

The four pillars of OOP

PillarSimple meaningReal-life example
EncapsulationHide internal details, show only what's neededAn ATM — you press buttons, you don't see the wiring
InheritanceChild gets properties from parentA Car inherits from Vehicle
PolymorphismSame action, different behaviorA dog and a cat both "speak" — differently
AbstractionShow only essential featuresA TV remote — press power, ignore the circuit

Key terms

  • Class — a template that defines properties and behaviors.
  • Object — an instance of a class.
  • Field / attribute — a variable inside a class (e.g. name, age).
  • Method — a function inside a class (e.g. bark(), eat()).
  • Constructor — a special method that runs when an object is created.
OOPIntro.java
public class Dog {
    String name;
    int age;

    // Constructor — runs when an object is created
    Dog(String name, int age) {
        this.name = name;
        this.age = age;
    }

    void bark() {
        System.out.println(name + " says: Woof!");
    }

    public static void main(String[] args) {
        Dog myDog = new Dog("Bruno", 3);
        Dog streetDog = new Dog("Rocky", 5);

        myDog.bark();      // Bruno says: Woof!
        streetDog.bark();  // Rocky says: Woof!
    }
}
Summary: OOP models real-world things in code. A class is a blueprint; an object is a real instance built from it. The four pillars — Encapsulation, Inheritance, Polymorphism, Abstraction — are explored one at a time starting from Module 04.
Recap from Module 01A class is a blueprint. An object is created using new. A constructor runs when an object is created. Now we go deeper into how classes actually work.

Anatomy of a class

ClassName.java
public class ClassName {
    // 1. Fields (state)
    dataType fieldName;

    // 2. Constructor
    ClassName(parameters) { ... }

    // 3. Methods (behavior)
    returnType methodName(parameters) { ... }
}

class Counter {
    int count;         // instance — each object has its own

Method overloading

Same method name, different parameters — Java picks the right one automatically.

Greeter.java
void greet() { ... }                       // no param
void greet(String name) { ... }            // one param
void greet(String name, int n) { ... }     // two params

The this keyword

this refers to the current object. It's used when a field name and a parameter name are the same.

Person.java
class Person {
    String name;

    Person(String name) {
        this.name = name;   // 'this.name' = field, 'name' = parameter
    }
}

Getters & setters

A good practice: keep fields private and provide public methods to read/write them.

BankAccount.java
class BankAccount {
    private double balance;   // hidden from outside

    public double getBalance() {           // getter — read
        return balance;
    }

    public void setBalance(double amount) { // setter — write
        if (amount >= 0) balance = amount;
    }
}
Covers: instance vs. static fields, method overloading, the this keyword, and getters/setters — with multiple objects sharing static data.
Recap from Module 02Fields store data, methods define behavior. this refers to the current object. Getters/setters protect private fields. Now: the special method that runs when an object is born.

A constructor has the same name as the class, has no return type (not even void), and runs automatically when you use new.

Four types of constructors

Cat.java
// 1. Default — Java adds this automatically if you write none at all
class Cat { String name; }
Cat c = new Cat(); // works even with no constructor written

// 2. No-arg — written by hand, sets default values
class Cat {
    String name;
    Cat() { name = "Unknown"; }
}

// 3. Parameterized — accepts values at creation
class Cat {
    String name;
    Cat(String name) { this.name = name; }
}
Cat c = new Cat("Whiskers");

// 4. Copy constructor — builds a new object from an existing one
class Cat {
    String name;
    Cat(Cat other) { this.name = other.name; }
}
Cat original = new Cat("Luna");
Cat copy = new Cat(original);

Constructor chaining with this()

One constructor can call another using this(), avoiding repeated code across multiple constructors.

Phone.java
class Phone {
    String brand, model;
    int storage;

    Phone() {
        this("Unknown", "Unknown", 64);      // calls 3-param constructor
    }

    Phone(String brand, String model) {
        this(brand, model, 128);             // calls 3-param constructor
    }

    Phone(String brand, String model, int storage) {
        this.brand = brand;
        this.model = model;
        this.storage = storage;
    }
}
Rule: this() must always be the first line inside a constructor.

Constructor vs. method

Summary: no-arg constructors set defaults, parameterized ones pass custom values, copy constructors duplicate an existing object, and chained (this()) constructors avoid repeated initialization code.
Recap from Module 03Constructors initialize objects. this() chains constructors to avoid repetition. A copy constructor creates an independent duplicate. Now: the first of the four OOP pillars.

Encapsulation means bundling data (fields) and behavior (methods) together inside a class, and hiding the internal details from the outside world — like a medicine capsule: the medicine is hidden inside, you only interact with the outer shell.

Access modifiers

ModifierSame classSame packageSubclassEverywhere
private✅❌❌❌
(default)✅✅❌❌
protected✅✅✅❌
public✅✅✅✅
Golden rule: fields → always private. Methods → public only when they need to be used from outside the class.

Getters and setters, with validation

Employee.java
class Employee {
    private String name;
    private int salary;

    public String getName() {          // getter — read only
        return name;
    }

    public void setSalary(int salary) { // setter — write, with validation
        if (salary >= 5000) {
            this.salary = salary;
        } else {
            System.out.println("Salary too low!");
        }
    }
}

Naming convention: field age → getter getAge(), setter setAge(). For a boolean field like active, the getter is isActive() instead of getActive().

Why encapsulation matters

  • Validation — reject invalid data before it's stored.
  • Read-only fields — provide a getter only, no setter, so a value set in the constructor can never change later (e.g. a national ID).
  • Computed values — a getter can calculate a value on the fly instead of storing it separately (e.g. getTotalSalary() = base + bonus).

Immutable classes

An object whose state can never change after creation. Rules: all fields private final, no setters, only getters, and every value set in the constructor.

ImmutablePoint.java
public final class ImmutablePoint {
    private final int x;
    private final int y;

    public ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int getX() { return x; }
    public int getY() { return y; }
    // No setters — once created, x and y never change
}

Java's own String class is immutable — that's why str.toUpperCase() returns a new String instead of modifying the original.

Covers: a basic Person example, validated Employee salary rules, a read-only NationalID, a computed ShoppingCart total, and an immutable Temperature class.
Recap from Module 04private fields hide data from outside. Setters validate before changing data. Immutable classes never change after creation. Now: the second OOP pillar.

Inheritance lets a class acquire the fields and methods of another class. The class being inherited from is the parent (superclass); the class that inherits is the child (subclass). A child gets everything the parent has, and can add its own behavior on top.

The extends keyword

Dog.java
class Animal {
    String name;
    void eat() { System.out.println(name + " is eating."); }
}

class Dog extends Animal {         // Dog inherits from Animal
    void bark() { System.out.println(name + " says: Woof!"); }
}

Dog d = new Dog();
d.name = "Bruno";
d.eat();   // inherited from Animal ✔
d.bark();  // Dog's own method ✔

The super keyword

Refers to the parent class — used to call the parent's constructor or a parent method it overrode.

Dog.java
class Animal {
    String name;
    Animal(String name) { this.name = name; }
    void describe() { System.out.println("Animal: " + name); }
}

class Dog extends Animal {
    String breed;
    Dog(String name, String breed) {
        super(name);              // calls Animal's constructor
        this.breed = breed;
    }

    @Override
    void describe() {
        super.describe();                          // runs Animal's version first
        System.out.println("Breed: " + breed);      // then adds more
    }
}
Rule: super() must always be the first line in a constructor.

Method overriding

A child class can redefine a method it inherited from the parent — same name, same parameters, same or wider access. @Override is recommended: it catches typos at compile time.

Inheritance chain

Java supports multi-level inheritance (but not multiple inheritance with classes):

chain.txt
Animal
 └── Mammal
      └── Dog

class Animal { ... }
class Mammal extends Animal { ... }  // child of Animal
class Dog extends Mammal { ... }     // child of Mammal — inherits from BOTH

The final keyword

Used onEffect
final classCannot be extended — no child classes
final methodCannot be overridden by a child
final fieldValue cannot change after assignment

What is not inherited

  • private fields and methods (hidden, though technically present)
  • Constructors (not inherited — called via super())
  • Static methods (belong to the class, not the object)
Covers: Animal → Dog/Cat/Bird, super for constructor and method calls, multi-level Vehicle → Car → ElectricCar, and a Person → Employee → Manager staff hierarchy.
Recap from Module 05extends gives a child all of the parent's fields and methods. super() calls the parent constructor. @Override redefines an inherited method. Now: the third OOP pillar.

Polymorphism means "many forms" — the same method name behaves differently depending on which object actually calls it.

Poly.java
Animal a = new Dog();
a.makeSound();  // prints "Woof!" — not "Some sound..."

Even though a is declared as Animal, Java runs the Dog version at runtime. That's polymorphism.

Two types

TypeAlso calledResolved atHow
Compile-timeStatic / overloadingCompile timeSame method name, different parameters
RuntimeDynamic / overridingRuntimeChild overrides parent's method
Printer.java
// Compile-time — overloading
class Printer {
    void print(int n)    { System.out.println("Int: " + n); }
    void print(double d) { System.out.println("Double: " + d); }
    void print(String s) { System.out.println("String: " + s); }
}

// Runtime — overriding
Animal a;
a = new Dog(); a.makeSound();  // Woof!
a = new Cat(); a.makeSound();  // Meow!
a = new Bird(); a.makeSound(); // Tweet!

Upcasting and downcasting

Casting.java
Animal a = new Dog("Bruno", 3);  // upcast — automatic, safe
a.eat();                        // works — inherited from Animal
// a.fetch();                   // compile error — Animal doesn't know fetch()

Dog d = (Dog) a;                // downcast — must cast explicitly
d.fetch();                      // now Dog-only methods are accessible
Careful: downcasting to the wrong type throws ClassCastException at runtime.

Checking with instanceof

InstanceofCheck.java
if (a instanceof Dog d) {   // check AND cast in one line (Java 16+)
    d.fetch();
}

Polymorphic arrays

Store different child objects in one parent-typed array — each still runs its own version:

Zoo.java
Animal[] zoo = {
    new Dog("Bruno", 3),
    new Cat("Luna", 2),
    new Bird("Sky", 1)
};

for (Animal a : zoo) {
    a.makeSound();  // each animal makes its OWN sound
}
Covers: overloading, overriding with a parent reference, safe up/downcasting, instanceof, a polymorphic shape-area calculator, and a real-world payment system.
Recap from Module 06Overloading resolves at compile time; overriding resolves at runtime. A parent reference can hold any child object. Check instanceof before downcasting. Now: the fourth and final OOP pillar.

Abstraction means hiding complex implementation and showing only the essential features — a TV remote: you press "power," you don't see the circuit. In Java, abstraction is achieved with abstract classes (this module) and interfaces (Module 08).

Abstract class

Cannot be instantiated. May contain abstract methods (declared, no body) alongside concrete methods (fully implemented). A child class must implement every abstract method.

Shape.java
abstract class Shape {
    abstract double area();       // no body — child MUST implement
    abstract double perimeter();  // no body — child MUST implement

    void describe() {             // concrete — has a body
        System.out.println("Area: " + area());
    }
}

class Circle extends Shape {
    double radius;
    Circle(double radius) { this.radius = radius; }
    double area() { return Math.PI * radius * radius; }
    double perimeter() { return 2 * Math.PI * radius; }
}

// Shape s = new Shape();  // compile error — cannot instantiate!
Shape s = new Circle(3);   // this works — Circle IS-A Shape
s.describe();

Abstract class rules

RuleDetail

Can have concrete methodsFully implemented
Can have constructorsCalled via super() from the child
Child must implement all abstract methodsOr the child must also be abstract

Covers: Shape → Circle/Rectangle/Triangle, an abstract Vehicle with a constructor, an Employee payroll system, and the Template Method Pattern — the parent defines the flow, the child fills in the steps.
Recap from Module 07An abstract class cannot be instantiated. Abstract methods have no body — the child must implement them. Now: Java's most powerful abstraction tool.

An interface is a pure contract — it defines what a class must do, but never how. A "Flyable" contract says "you must be able to fly"; a Bird, a Plane, and a Drone can all follow it, each flying completely differently.

Flyable.java
interface Flyable {
    int MAX_SPEED = 100;        // implicitly public static final (a constant)

    void fly();                 // implicitly public abstract — implementer MUST provide this

    default void greet() {      // has a body — optional to override (Java 8+)
        System.out.println("Ready for takeoff.");
    }

    static void info() {        // called on the interface itself
        System.out.println("This is a Flyable contract.");
    }
}

class Bird implements Flyable { public void fly() { System.out.println("Flap flap"); } }
class Plane implements Flyable { public void fly() { System.out.println("Engines on"); } }
Note: interface methods are public by default — your implementation must also be public.

Multiple interfaces — the key advantage

A class can implement several interfaces at once, which is how Java works around single inheritance for classes:

Duck.java
interface Swimmable { void swim(); }
interface Runnable  { void run(); }
interface Flyable   { void fly(); }

class Duck implements Swimmable, Runnable, Flyable {
    public void swim() { System.out.println("Duck swims!"); }
    public void run()  { System.out.println("Duck runs!"); }
    public void fly()  { System.out.println("Duck flies!"); }
}

Interface vs. abstract class

Use when"This object can do X""This object is a type of X"

Functional interfaces & lambdas (Java 8+)

An interface with exactly one abstract method can be implemented inline with a lambda:

Calculator.java
@FunctionalInterface
interface Calculator {
    int calculate(int a, int b);
}

Calculator add = (a, b) -> a + b;   // lambda instead of a whole class!
Calculator mul = (a, b) -> a * b;

System.out.println(add.calculate(5, 3));  // 8
System.out.println(mul.calculate(5, 3));  // 15
Covers: a basic Drawable interface, Duck implementing three interfaces at once, polymorphism through an interface type, default methods, functional interfaces with lambdas, and a real-world Smart Home device system.

S — Single Responsibility

If a class both calculates an invoice and emails it, a change to the email format now risks breaking the invoice math. Split them:

SingleResponsibility.java
class InvoiceCalculator {
    double total(double[] items) {
        double sum = 0;
        for (double i : items) sum += i;
        return sum;
    }
}

class InvoiceMailer {
    void send(String to, double total) {
        System.out.println("Emailing $" + total + " invoice to " + to);
    }
}

O — Open/Closed

Adding a new rule should never require editing code that already works. Below, a new discount type never touches PriceCalculator.

OpenClosed.java
interface DiscountRule {
    double apply(double price);
}

class StudentDiscount implements DiscountRule {
    public double apply(double price) { return price * 0.9; }
}

// Adding SeasonalDiscount later needs zero changes below:
class PriceCalculator {
    private DiscountRule rule;
    PriceCalculator(DiscountRule rule) { this.rule = rule; }
    double finalPrice(double price) { return rule.apply(price); }
}

L — Liskov Substitution

A classic trap: Square extends Rectangle looks reasonable, but if setting a Square's width silently changes its height too, code written for Rectangle breaks when handed a Square. If a subclass can't honor the parent's promises, it shouldn't be a subclass.

I — Interface Segregation

A fat Worker interface with work() and eat() forces a RobotWorker to implement eat() with nonsense. Split it:

InterfaceSegregation.java
interface Workable { void work(); }
interface Eatable  { void eat(); }

class Human implements Workable, Eatable {
    public void work() { System.out.println("Human working"); }
    public void eat()  { System.out.println("Human eating"); }
}

class Robot implements Workable {   // no forced eat()
    public void work() { System.out.println("Robot working"); }
}

D — Dependency Inversion

A high-level class shouldn't depend on a concrete low-level class — both should depend on an interface. This is what lets you swap a real database for a test double without touching business logic.

DependencyInversion.java
interface MessageSender {
    void send(String message);
}

class EmailSender implements MessageSender {
    public void send(String message) { System.out.println("Email: " + message); }
}

class OrderService {
    private MessageSender sender;          // depends on the interface, not EmailSender
    OrderService(MessageSender sender) { this.sender = sender; }
    void confirm() { sender.send("Order confirmed"); }
}
Why it matters: none of these are rules to follow blindly — each one answers "what breaks when this needs to change?" SOLID is the bridge between the four pillars and the design patterns in Module 10, which are really just SOLID applied to recurring problems.
Recap from Module 09SOLID keeps classes focused, extensible, substitutable, lean, and loosely coupled. Design patterns are named, battle-tested solutions built on top of those same ideas — a shared vocabulary for problems every codebase eventually runs into.

The Gang-of-Four patterns fall into three families, grouped by the kind of problem they solve.

Creational — how objects get created

PatternSolves
SingletonGuarantee exactly one instance, with one global access point
FactoryHide new behind a method, so callers don't pick the concrete class
BuilderConstruct a complex object step by step, avoiding ten-parameter constructors
NotificationFactory.java
interface Notification { void send(String msg); }

class EmailNotification implements Notification {
    public void send(String msg) { System.out.println("Email: " + msg); }
}
class SmsNotification implements Notification {
    public void send(String msg) { System.out.println("SMS: " + msg); }
}

class NotificationFactory {
    static Notification create(String type) {
        return switch (type) {
            case "email" -> new EmailNotification();
            case "sms"   -> new SmsNotification();
            default -> throw new IllegalArgumentException("Unknown type: " + type);
        };
    }
}

Notification n = NotificationFactory.create("sms");
n.send("Your order has shipped");  // SMS: Your order has shipped

Structural — how classes are wired together

PatternSolves
AdapterBridge two incompatible interfaces
DecoratorAttach new behavior at runtime by wrapping, instead of subclassing every combination
CompositeTreat a single object and a group of objects the same way (tree structures)
CoffeeDecorator.java
interface Coffee { double cost(); String describe(); }

class SimpleCoffee implements Coffee {
    public double cost() { return 2.0; }
    public String describe() { return "Coffee"; }
}

abstract class CoffeeDecorator implements Coffee {
    protected Coffee inner;
    CoffeeDecorator(Coffee inner) { this.inner = inner; }
}

class MilkDecorator extends CoffeeDecorator {
    MilkDecorator(Coffee inner) { super(inner); }
    public double cost() { return inner.cost() + 0.5; }
    public String describe() { return inner.describe() + " + Milk"; }
}

Coffee order = new MilkDecorator(new SimpleCoffee());
System.out.println(order.describe() + " = $" + order.cost()); // Coffee + Milk = $2.5

Behavioral — how objects communicate

PatternSolves
ObserverSubscribe to another object's events — one change notifies every listener
StrategyMake an algorithm swappable at runtime
CommandWrap a request as an object, so it can be queued, logged, or undone
WeatherStation.java
import java.util.ArrayList;
import java.util.List;

interface Observer { void update(double temperature); }

class WeatherStation {
    private List<Observer> observers = new ArrayList<>();
    void subscribe(Observer o) { observers.add(o); }
    void setTemperature(double t) {
        for (Observer o : observers) o.update(t);
    }
}

class PhoneDisplay implements Observer {
    public void update(double t) { System.out.println("Phone shows: " + t + "°C"); }
}

WeatherStation station = new WeatherStation();
station.subscribe(new PhoneDisplay());
station.setTemperature(31.5);  // Phone shows: 31.5°C
Takeaway: no pattern here was used for its own sake — each one answers a specific "what happens when this changes?" question the design would otherwise run into. That's the same question every module in this course has been building toward.
