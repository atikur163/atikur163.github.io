MODULE 01

Classes & Objects

A class is a blueprint — it describes what data an object holds (its fields) and what it can do (its methods). An object is one concrete thing built from that blueprint. A Car class might describe that every car has a color and a speed and can accelerate; the specific red car sitting in memory at runtime is the object.

Every object you create from the same class shares the same structure but keeps its own copy of the data. That's the whole idea of OOP in one sentence: bundle data and the behavior that acts on it into a single unit.

Car.java
public class Car {
    private String color;
    private int speed;

    public Car(String color) {
        this.color = color;
        this.speed = 0;
    }

    public void accelerate(int amount) {
        this.speed += amount;
    }

    public String describe() {
        return color + " car moving at " + speed + " km/h";
    }

    public static void main(String[] args) {
        Car red = new Car("Red");
        Car blue = new Car("Blue");

        red.accelerate(40);
        blue.accelerate(15);

        System.out.println(red.describe());   // Red car moving at 40 km/h
        System.out.println(blue.describe());  // Blue car moving at 15 km/h
    }
}
Note: red and blue are two separate objects built from one Car class — each keeps its own speed.
MODULE 02

Encapsulation

Encapsulation means hiding an object's internal data and only exposing it through controlled methods. In practice: mark fields private, then provide public get/set methods — or better, methods named after what they actually do — so outside code can never leave the object in an invalid state.

Without encapsulation, any code anywhere could set a bank balance to a negative number. With it, the class itself is the only place that's allowed to make that decision, so it can enforce the rule every time.

BankAccount.java
public class BankAccount {
    private double balance;

    public BankAccount(double openingBalance) {
        this.balance = openingBalance;
    }

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Deposit must be positive");
        balance += amount;
    }

    public void withdraw(double amount) {
        if (amount > balance) throw new IllegalStateException("Insufficient funds");
        balance -= amount;
    }

    public double getBalance() {
        return balance;
    }

    public static void main(String[] args) {
        BankAccount acc = new BankAccount(100);
        acc.deposit(50);
        acc.withdraw(30);
        System.out.println(acc.getBalance()); // 120.0
        // acc.balance = -500;  // not possible — the field is private
    }
}
Note: balance can never be touched directly from outside the class — every change goes through a method that can validate it first.
MODULE 03

Inheritance

Inheritance lets one class reuse the fields and methods of another. A Car and a Truck are both vehicles — instead of writing the shared logic twice, both extend a common Vehicle class and add only what's specific to them.

Use inheritance for a genuine "is-a" relationship (a Truck is a Vehicle). If the relationship is really "has-a" (a Car has a Engine), composition — module 8 — is usually the better fit.

Vehicle.java
public class Vehicle {
    protected String model;
    protected int speed = 0;

    public Vehicle(String model) {
        this.model = model;
    }

    public void accelerate(int amount) {
        speed += amount;
    }

    public String status() {
        return model + " at " + speed + " km/h";
    }
}

public class Truck extends Vehicle {
    private int cargoKg;

    public Truck(String model, int cargoKg) {
        super(model);          // reuse Vehicle's constructor
        this.cargoKg = cargoKg;
    }

    public String status() {
        return super.status() + ", carrying " + cargoKg + "kg";
    }

    public static void main(String[] args) {
        Truck t = new Truck("Volvo FH", 2000);
        t.accelerate(60);
        System.out.println(t.status()); // Volvo FH at 60 km/h, carrying 2000kg
    }
}
Note: Truck gets accelerate() for free from Vehicle, and only overrides status() to add its own detail.
MODULE 04

Polymorphism

Polymorphism means the same method call behaves differently depending on the actual object it's called on. If Circle and Square both extend Shape and override area(), you can hold a list of Shape and call area() on each without ever checking what specific shape it is — Java figures out which version to run at runtime.

This is what lets you add a brand-new shape later without touching any code that already works with Shape.

Shapes.java
public abstract class Shape {
    public abstract double area();
}

public class Circle extends Shape {
    private double radius;
    public Circle(double radius) { this.radius = radius; }
    public double area() { return Math.PI * radius * radius; }
}

public class Square extends Shape {
    private double side;
    public Square(double side) { this.side = side; }
    public double area() { return side * side; }
}

public class ShapeDemo {
    public static void main(String[] args) {
        Shape[] shapes = { new Circle(3), new Square(4) };

        for (Shape s : shapes) {
            // same call, different behavior per actual object
            System.out.printf("%.2f%n", s.area());
        }
        // 28.27
        // 16.00
    }
}
Note: The loop never checks "is this a Circle or a Square?" — each object knows how to compute its own area().
MODULE 05

Abstraction — Abstract Classes & Interfaces

Abstraction means exposing only what a caller needs to know, and hiding how it's actually done. Java gives you two tools for this: an abstract class (a partial blueprint that can hold shared code and force subclasses to fill in the rest) and an interface (a pure contract — just a list of methods a class promises to implement, with no shared state).

Rule of thumb: use an abstract class when related classes share real code; use an interface when unrelated classes just need to promise the same capability — for example, both a Duck and a Plane can implement Flyable even though they share nothing else.

Flyable.java
public interface Flyable {
    void fly();
}

public class Duck implements Flyable {
    public void fly() { System.out.println("Duck flaps its wings"); }
}

public class Plane implements Flyable {
    public void fly() { System.out.println("Plane fires its engines"); }
}

public class SkyDemo {
    public static void main(String[] args) {
        Flyable[] fliers = { new Duck(), new Plane() };
        for (Flyable f : fliers) {
            f.fly();
        }
        // Duck flaps its wings
        // Plane fires its engines
    }
}
Note: Duck and Plane share no parent class and no code — only the promise that they can both fly().
MODULE 06

SOLID Principles

SOLID is five rules of thumb for keeping object-oriented code easy to change without breaking it.

  • S — Single Responsibility: a class should have exactly one reason to change.
  • O — Open/Closed: open for extension, closed for modification — add new behavior without editing existing, working code.
  • L — Liskov Substitution: a subclass must be usable anywhere its parent class is expected, without breaking anything.
  • I — Interface Segregation: prefer several small, specific interfaces over one giant one that forces classes to implement methods they don't need.
  • D — Dependency Inversion: depend on interfaces, not concrete classes, so pieces can be swapped independently.

The example below shows Open/Closed in action: adding a new discount rule never touches PriceCalculator.

OpenClosedExample.java
public interface DiscountRule {
    double apply(double price);
}

public class NoDiscount implements DiscountRule {
    public double apply(double price) { return price; }
}

public class StudentDiscount implements DiscountRule {
    public double apply(double price) { return price * 0.9; }
}

// Adding SeasonalDiscount later needs zero changes here:
public class PriceCalculator {
    private DiscountRule rule;
    public PriceCalculator(DiscountRule rule) { this.rule = rule; }

    public double finalPrice(double price) {
        return rule.apply(price);
    }

    public static void main(String[] args) {
        PriceCalculator calc = new PriceCalculator(new StudentDiscount());
        System.out.println(calc.finalPrice(1000)); // 900.0
    }
}
MODULE 07

Creational Patterns — Singleton, Factory, Builder

Creational patterns deal with one problem: how objects get created, so the rest of the code doesn't have to know the messy details.

  • Singleton guarantees a class has exactly one instance, with one global access point — useful for something like an application-wide configuration.
  • Factory hides the new keyword behind a method, so callers ask for "a Notification" without knowing or caring whether they get an EmailNotification or SmsNotification.
  • Builder constructs a complex object step by step, which avoids constructors with ten optional parameters.
Singleton.java
public class AppConfig {
    private static AppConfig instance;
    private String environment;

    private AppConfig() { this.environment = "production"; }

    public static AppConfig getInstance() {
        if (instance == null) {
            instance = new AppConfig();
        }
        return instance;
    }

    public String getEnvironment() { return environment; }
}
NotificationFactory.java
public interface Notification {
    void send(String message);
}

public class EmailNotification implements Notification {
    public void send(String message) { System.out.println("Email: " + message); }
}

public class SmsNotification implements Notification {
    public void send(String message) { System.out.println("SMS: " + message); }
}

public class NotificationFactory {
    public static Notification create(String type) {
        return switch (type) {
            case "email" -> new EmailNotification();
            case "sms" -> new SmsNotification();
            default -> throw new IllegalArgumentException("Unknown type: " + type);
        };
    }

    public static void main(String[] args) {
        Notification n = NotificationFactory.create("sms");
        n.send("Your order has shipped"); // SMS: Your order has shipped
    }
}
MODULE 08

Structural Patterns — Adapter, Decorator, Composite

Structural patterns deal with how classes and objects are wired together into larger structures.

  • Adapter converts one class's interface into another one a client expects — a bridge between incompatible interfaces.
  • Decorator attaches new behavior to an object at runtime by wrapping it, instead of creating a new subclass for every combination of features.
  • Composite lets you treat a single object and a group of objects the same way — useful for tree structures like a file system.

The example below decorates a plain coffee with add-ons, each wrapping the last.

CoffeeDecorator.java
public interface Coffee {
    double cost();
    String describe();
}

public class SimpleCoffee implements Coffee {
    public double cost() { return 2.0; }
    public String describe() { return "Coffee"; }
}

public abstract class CoffeeDecorator implements Coffee {
    protected Coffee inner;
    public CoffeeDecorator(Coffee inner) { this.inner = inner; }
}

public class MilkDecorator extends CoffeeDecorator {
    public MilkDecorator(Coffee inner) { super(inner); }
    public double cost() { return inner.cost() + 0.5; }
    public String describe() { return inner.describe() + " + Milk"; }
}

public class SyrupDecorator extends CoffeeDecorator {
    public SyrupDecorator(Coffee inner) { super(inner); }
    public double cost() { return inner.cost() + 0.7; }
    public String describe() { return inner.describe() + " + Syrup"; }
}

public class CafeDemo {
    public static void main(String[] args) {
        Coffee order = new SyrupDecorator(new MilkDecorator(new SimpleCoffee()));
        System.out.println(order.describe() + " = $" + order.cost());
        // Coffee + Milk + Syrup = $3.2
    }
}
MODULE 09

Behavioral Patterns — Observer, Strategy, Command

Behavioral patterns deal with how objects communicate and share responsibility.

  • Observer lets objects subscribe to another object's events, so a change in one place automatically notifies everyone who's listening.
  • Strategy makes an algorithm swappable — a class holds a reference to a strategy interface, and the specific algorithm is plugged in from outside.
  • Command wraps a request as an object, so it can be queued, logged, or undone.

Below, an Observer example: a WeatherStation notifies every subscribed display when the temperature changes.

WeatherStation.java
import java.util.ArrayList;
import java.util.List;

public interface Observer {
    void update(double temperature);
}

public class WeatherStation {
    private List<Observer> observers = new ArrayList<>();

    public void subscribe(Observer o) { observers.add(o); }

    public void setTemperature(double temp) {
        for (Observer o : observers) {
            o.update(temp);
        }
    }
}

public class PhoneDisplay implements Observer {
    public void update(double temperature) {
        System.out.println("Phone shows: " + temperature + "°C");
    }
}

public class WeatherDemo {
    public static void main(String[] args) {
        WeatherStation station = new WeatherStation();
        station.subscribe(new PhoneDisplay());
        station.setTemperature(31.5); // Phone shows: 31.5°C
    }
}
MODULE 10

Case Study — Designing a Small System with OOP

This module puts modules 1–9 together in one small, coherent design: a food-delivery order system. The goal isn't more syntax — it's showing how the pieces are chosen, not just used.

  • Encapsulation keeps an Order's total in charge of its own math.
  • Abstraction via a PaymentMethod interface hides how each payment type actually charges the customer.
  • Strategy lets the payment method be swapped per order, at runtime.
  • Factory creates the right PaymentMethod from a plain string, so the rest of the code never touches new directly.
OrderSystem.java
import java.util.ArrayList;
import java.util.List;

public interface PaymentMethod {
    void pay(double amount);
}

public class CardPayment implements PaymentMethod {
    public void pay(double amount) { System.out.println("Charged $" + amount + " to card"); }
}

public class CashOnDelivery implements PaymentMethod {
    public void pay(double amount) { System.out.println("Collect $" + amount + " on delivery"); }
}

public class PaymentFactory {
    public static PaymentMethod create(String type) {
        return switch (type) {
            case "card" -> new CardPayment();
            case "cod"  -> new CashOnDelivery();
            default -> throw new IllegalArgumentException("Unknown payment type");
        };
    }
}

public class Order {
    private List<Double> itemPrices = new ArrayList<>();
    private PaymentMethod payment;

    public Order(String paymentType) {
        this.payment = PaymentFactory.create(paymentType);
    }

    public void addItem(double price) { itemPrices.add(price); }

    public double total() {
        double sum = 0;
        for (double p : itemPrices) sum += p;
        return sum;
    }

    public void checkout() {
        payment.pay(total());
    }

    public static void main(String[] args) {
        Order order = new Order("card");
        order.addItem(12.5);
        order.addItem(7.25);
        order.checkout(); // Charged $19.75 to card
    }
}
Takeaway: none of these patterns were used for their own sake — each one solved a specific "what happens when this changes?" question the design would otherwise run into.
